# `claude-code-sourcemap` 公众号三连发大纲

## 系列定位
一组围绕 `claude-code-sourcemap` 的解析文章，按“认知破冰 -> 使用理解 -> 工程拆解”递进。

## 发布顺序
1. 第一篇：认知破冰篇
2. 第二篇：AI Coding 理解篇
3. 第三篇：Agent CLI 架构篇

## 第一篇
### 暂定标题
有人把 Claude Code“拆”出来了：我顺着 source map 看到了什么

### 开篇钩子
“一个 source map，为什么能让这么多程序员兴奋？”

### 主体结构
1. 这个仓库到底是什么，不是什么
2. 为什么 source map 还原会让开发者上头
3. 目录结构像一张产品剖面图
4. 我们到底能从中看到什么，又不能脑补什么
5. 预告下一篇：为什么这会改变我对 AI Coding 的理解

## 第二篇
### 暂定标题
看完这份还原仓库，我对 AI Coding 工具的理解变了

### 开篇钩子
“AI Coding 工具真正难的，也许不是写代码，而是把工作流组织起来。”

### 主体结构
1. 不要把 AI Coding 工具理解成聊天框
2. 包描述与目录结构暴露出的真实能力模型
3. `commands`、`tools`、`skills`、`plugins` 各自意味着什么
4. 重度用户真正该优化的是任务拆分、上下文与验证闭环
5. 预告下一篇：如果你自己做 Agent CLI，该如何看这份仓库

## 第三篇
### 暂定标题
如果你要做一个 AI Agent CLI，这份仓库最值得拆的不是模型，而是工程组织

### 开篇钩子
“很多团队以为自己在做模型接入，实际上他们在做的是复杂得多的工程系统。”

### 主体结构
1. 从入口、命令、工具到服务的分层启发
2. 为什么 `context`、`coordinator`、`plugins`、`skills` 很关键
3. `remote`、`voice`、`vim` 暗示了什么产品野心
4. 做 Agent CLI 时最容易踩的工程坑
5. 结尾：如何理性看待这类“还原源码”

## 每篇统一写法
- 开头先给一个能抓人的问题或判断
- 中间穿插对 README 和包描述的解释
- 不堆源码，不做“技术考古炫技”
- 结尾保留系列导流和个人判断
