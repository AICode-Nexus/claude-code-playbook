# `claude-code-sourcemap` 公众号三连发选题分析

## 选题
基于 `https://github.com/ChinaSiro/claude-code-sourcemap`，撰写一组适合微信公众号发布的三篇系列解析文章。

## 素材事实
- 仓库为**非官方**整理版，基于公开 npm 包与 source map 分析还原，README 明确标注“仅供研究使用”，不代表官方内部仓库结构。
- 对应 npm 包为 `@anthropic-ai/claude-code`，还原版本为 `2.1.88`。
- README 声称还原出 `4756` 个文件，其中 `1884` 个为 `.ts`/`.tsx`。
- README 列出的目录包括 `tools`、`commands`、`services`、`context`、`coordinator`、`assistant`、`plugins`、`skills`、`voice`、`vim` 等。
- 包描述强调该 CLI 可以“理解代码库、编辑文件、运行终端命令、处理完整工作流”。

## 目标读者
### 读者一：技术爱好者 / 泛程序员
- 关注点：这仓库为什么值得看，source map 为什么能还原出这么多东西。
- 阅读期待：获得“原来 AI Coding 工具背后长这样”的认知震撼。

### 读者二：AI Coding 重度用户
- 关注点：这份仓库如何帮助重新理解 AI Coding 的能力边界和正确用法。
- 阅读期待：从“提示词技巧”升级到“工作流理解”。

### 读者三：做工程平台 / CLI / Agent 的开发者
- 关注点：这个还原仓库暴露出怎样的工程组织方式与模块边界。
- 阅读期待：拿到可迁移的工程启发，而不是八卦式围观。

## 内容目标
- 做成一组“由浅入深”的三连发，而不是三篇平行稿。
- 用技术博主型解析稿为主、开发者一手观察稿为辅。
- 保留话题性，但避免耸动标题和越界解读。
- 每篇结尾自然导流下一篇，形成连载感。

## 成功标准
- 清晰度：`9/10`
- 技术准确性：`9/10`
- 可读性与传播性：`8.5/10`
- 观点密度：`8.5/10`
- 微信公众号适配度：`9/10`

## 输出形式
- 三篇 Markdown 正文
- 每篇附带：
  - 标题
  - 副标题
  - 摘要
  - 封面文案

## 风险与边界
- 不能把仓库写成“内部源码泄露”。
- 不能把 README 中的目录结构等同于 Anthropic 官方内部真实仓库结构。
- 不能把还原版本 `2.1.88` 写成“Claude Code 当前全部真实实现”。
- 所有判断都应建立在公开材料之上，并明确“研究用途、非官方”前提。
